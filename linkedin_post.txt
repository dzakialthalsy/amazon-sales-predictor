🚀 From Raw Data to Deployed ML Model: My End-to-End Amazon Sales Prediction Journey

I'm excited to share my latest Machine Learning project where I built and 
deployed a sales prediction system for e-commerce products on Amazon. This 
project took me through the complete ML lifecycle - from messy raw data to 
a production-ready web application.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📊 PHASE 1: DATA COLLECTION & EXPLORATION

The journey started with 7 different CSV files from various sources:
• Sales reports from multiple time periods
• Amazon sale reports with 700K+ rows
• International sales data
• Profit & Loss statements
• Expense reports

Total: 703,966 rows of raw transaction data with 32 features including SKU 
codes, product categories, sizes, colors, shipping details, and sales amounts.

The data was messy - mixed types, missing values, inconsistent formats. 
Real-world data engineering at its finest! 🔧

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🧹 PHASE 2: DATA PREPROCESSING & FEATURE ENGINEERING

Key challenges addressed:
✓ Merged multiple data sources into a unified dataset
✓ Handled missing values across 9+ columns
✓ Encoded categorical variables (Size, Color, Category, State)
✓ Created meaningful features from raw transaction data
✓ Applied StandardScaler for feature normalization

Final feature set (7 features):
• Stock - Inventory levels
• index_sales_int - Sales performance index
• PCS - Pieces/Units sold
• RATE - Product price rate
• GROSS AMT - Total transaction amount
• Qty - Order quantity
• ship-postal-code - Geographic indicator

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🤖 PHASE 3: MODEL DEVELOPMENT & TRAINING

I experimented with multiple algorithms:
• Linear Regression (baseline)
• Decision Tree Regressor
• Random Forest Regressor ⭐
• Gradient Boosting Regressor
• CatBoost
• XGBoost
• LightGBM

After extensive hyperparameter tuning, Random Forest Regressor emerged as 
the champion with:
• max_depth: 47
• n_estimators: 88
• min_samples_split: 5
• random_state: 42

The model was trained on 80% of the data (563K samples) and tested on 20% 
(140K samples).

🌐 PHASE 5: WEB DEPLOYMENT

A model is only useful if people can use it. I built a complete web application:

Tech Stack:
• Backend: Flask (Python web framework)
• ML: scikit-learn, joblib
• Model Storage: Hugging Face Hub (895MB model file)
• Frontend: HTML5, CSS3, modern responsive design

Features:
✓ User-friendly input form for all 7 features
✓ Real-time prediction with instant results
✓ RESTful API endpoint for programmatic access
✓ Auto-download model from Hugging Face on first run
✓ Production-ready with Gunicorn WSGI server

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

💡 KEY CHALLENGES & SOLUTIONS

Challenge 1: Model file size (895MB) exceeded GitHub's 100MB limit
→ Solution: Used Hugging Face Hub for model hosting with auto-download 
   functionality

Challenge 2: Multiple data sources with inconsistent formats
→ Solution: Built robust data pipeline with proper validation and error 
   handling

Challenge 3: Feature scaling for accurate predictions
→ Solution: Implemented StandardScaler during training, reverse scaling in 
   production

🎯 WHAT I LEARNED

1. Real-world data is never clean - expect to spend 70% of your time on 
   preprocessing

2. Model deployment is just as important as model accuracy

3. Large model files require creative storage solutions (shoutout to 
   Hugging Face!)

4. End-to-end projects teach you 10x more than just modeling tutorials

5. Documentation and clean code matter - future you will thank present you
